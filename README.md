# Заняття 11 · Домашнє завдання

## Personal Finance Crew — Multi-Agent Orchestration

Розробити multi-agent систему **"Personal Finance Coach"** — команду спеціалізованих агентів, яка відповідає на фінансові запити користувача. Виконати порівняння з single-agent baseline та підготувати звіт із даними: у яких випадках multi-agent архітектура переважає, у яких поступається, та чи виправдане ускладнення.

Ролі агентів, набір інструментів, структура коду, golden set, формат звіту визначаються самостійно. Це є частиною завдання. Мінімальна вимога до crew — щонайменше 3 спеціалізовані агенти.

> **Примітка щодо проблем у процесі виконання.** У разі виникнення труднощів рекомендується дослідити проблему через документацію, GitHub issues та Stack Overflow. Якщо вирішення знайти не вдається, у звіті слід зафіксувати, що саме не вдалося реалізувати та з якої причини — це є цінним результатом, що демонструє розуміння обмежень інструментів. Допускається використання AI-асистентів (Claude, ChatGPT, Cursor) для діагностики помилок. Реальна інженерна робота переважно складається з debugging та пошуку обхідних шляхів — цей досвід є важливішим за бездоганне виконання за готовим керівництвом.

## Бізнес-контекст

**Замовник** — fintech-стартап, що розробляє AI-помічника для мобільного банківського застосунку. Існуюча вкладка "Аналітика" з графіками демонструє низький рівень engagement. Гіпотеза продукту: розмовний помічник із природньомовним інтерфейсом підвищить використання аналітичних функцій та retention.

**Бізнес-цілі:**

- Час відповіді на запит — не більше 10 секунд
- Cost per query — економічно прийнятний для бізнес-моделі
- Поради повинні призводити до дій користувача (наприклад, відписка від невикористовуваних підписок)

---

## Користувацькі сценарії

Помічник повинен якісно обробляти три категорії запитів.

### 1. Статистика та факти про витрати

> "Скільки витратив на каву минулого тижня?"
> "Топ-5 категорій витрат за червень?"
> "Дата останнього платежу за Netflix?"

Очікується: конкретна сума, коректний часовий період, мінімальна затримка.

### 2. Поради щодо економії на основі даних користувача

> "Де можна зекономити $200 цього місяця?"
> "На які підписки витрачається найбільше та чи всі вони необхідні?"
> "Як швидше виплатити кредитну картку?"

Поради повинні базуватися на конкретних числах із транзакцій користувача (наприклад: "$180 на Glovo, з яких 60% — після 21:00") та містити actionable-крок, який можна виконати негайно. Загальні рекомендації типу "consider reducing dining out" неприйнятні.

### 3. Multi-step аналіз і синтез

> "Порівняти витрати поточного року з минулорічними та визначити зміни"
> "Чи буде поточний місяць закрито у плюс?"
> "Якщо зменшити витрати на доставку вдвічі — яка економія за рік?"

Очікується: декілька запитів до даних, синтез результатів, висновок із числами.

## Вимоги до відповідей агента

- **Тон агента у відповідях.** Дружній, на "ти", без менторства. Емпатичне формулювання у стресових темах (борги, fraud).
- **Обґрунтованість.** Числа у відповідях беруться виключно з реальних даних користувача. Hallucinations неприпустимі.
- **Multi-turn контекст.** Послідовність "Скільки на каву?" → "$45" → "А місяць?" повинна коректно інтерпретуватись (другий запит стосується кави).
- **Escalation.** Запити щодо fraud або підозрілих транзакцій не вирішуються агентом самостійно — користувач направляється до служби підтримки.
- **Out of scope.** Запити поза скоупом (наприклад, "купи акції") відхиляються з ввічливим поясненням та переадресацією на доступні функції.

## Дані

У директорії `starter/data/` надано готовий dataset — використовуйте його як вхідні дані для агента. Самостійна генерація даних не вимагається.

**Файли:**

- `starter/data/transactions.csv` — 842 транзакції за 12 місяців (грудень 2024 — листопад 2025)
- `starter/data/generate.py` — скрипт генерації (для прозорості; за потреби можна модифікувати параметри та перегенерувати)

**Поля CSV:**

- `date` — ISO timestamp (з годиною)
- `merchant` — назва продавця
- `amount` — сума (від'ємна для витрат, додатна для salary)
- `currency` — `USD`
- `category` — `coffee` / `groceries` / `restaurants` / `delivery` / `transport` / `entertainment` / `shopping` / `health` / `subscriptions` / `utilities` / `salary` / `credit_payment` / `travel`
- `account` — `main_debit` або `credit_card`
- `recurring` — `True` / `False`

**Патерни в даних** (агент повинен їх знаходити):

- **Forgotten subscription:** `Sportlife` $15/міс, остання транзакція ~4 місяці тому
- **Late-night delivery:** ~50% delivery-замовлень (Glovo, Bolt Food, Uber Eats) — після 21:00
- **Weekend spike:** середня сума транзакції у вихідні на ~50% вища за будні
- **Coffee ritual:** $80–95/міс стабільно, переважно у будні зранку
- **Credit card behavior:** 2 з 3 місяців — minimum payment $50, 1 з 3 — повна виплата
- **Suspicious foreign txns:** 2 транзакції на credit card (Booking.com, AliExpress) — для тестування fraud-escalation сценарію

**Регенерація** (опціонально):

```bash
python3 starter/data/generate.py
```

Seed зафіксований (`random.seed(42)`) — повторний запуск дає той самий результат. Параметри генерації — у верхній частині `generate.py`.

> 👤 скільки витратила на каву минулого тижня

> 🤖 $34. Це 11 транзакцій — переважно Aroma Kava і Lviv Croissants, у будні зранку. На $7 більше ніж тиждень тому.

> 👤 чимало. де можу зекономити $200 цього місяця?

> 🤖 На основі даних за останні 3 місяці виділяю 3 реалістичні варіанти:
>
> **1. Доставка їжі — $180/міс (Glovo, Bolt Food).** 60% замовлень після 21:00 — імпульсні. Зменшення вдвічі = ~$90.
>
> **2. Підписки $47/міс.** Sportlife $15/міс — остання транзакція 4 місяці тому, ймовірно забута підписка. Рекомендується перевірити. −$15/міс.
>
> **3. Кава поза домом — $142/міс.** Не пропоную відмовлятися повністю, але приготування вдома 2 дні на тиждень = ~$40.
>
> Загалом ~$145. До цільових $200 не вистачає — потребується глибше скорочення в одній з категорій або додаткові статті. З чого почати?

> 👤 на моїй карті $890 в Booking.com 3 грудня, я не робила цю транзакцію

> 🤖 Імовірний fraud. Блокування картки та оформлення chargeback виходять за межі моїх можливостей — це робить служба підтримки. Рекомендовані дії:
>
> 1. Заблокувати картку: Картки → ця карта → Заблокувати
> 2. Звернутися до служби підтримки через чат застосунку — у них окрема процедура для disputed transactions
>
> За потреби можу показати останні 5 транзакцій по цій картці для виявлення інших підозрілих операцій.

## Метрики у звіті

Метрики збираються через LangSmith Experiments (більшість — автоматично з traces). У звіті повинні бути присутні наступні групи метрик для обох архітектур (crew та baseline):

**Performance (LangSmith збирає автоматично з traces):**

- `latency_p50` / `latency_p95` (мс)
- `cost_per_task` ($)
- `tokens_per_task`

**Якість відповідей (потребує custom evaluators):**

- `success_rate` — частка задач, що пройшли перевірку judge LLM
- `tool_selection_accuracy` — коректність вибору інструментів
- `groundedness` — частка числових значень у відповідях, що відповідають реальним даним

**Multi-agent specific (для crew, через tagging агентів у traces):**

- `inter_agent_overhead_pct` — частка токенів, витрачена на context-passing між агентами
- `cost_breakdown_by_agent` — розподіл вартості за агентами

## UI (Streamlit)

Простий Streamlit-застосунок для demo:

- Поле введення запиту та кнопка Submit
- Відображення відповіді
- Бічна панель: вибір архітектури (`crew` / `baseline`), опціонально — вибір моделі
- Розгортуваний блок "Trace" під відповіддю: послідовність викликів агентів та інструментів, latency, cost
- Вкладка "Eval": запуск golden set, таблиця результатів обох архітектур

Не вимагається: автентифікація, persistent users, дизайн. Призначення — dev tool для demo.

## Критерії оцінки

- Корисність відповідей: числа з реальних даних, actionable-формулювання, відсутність generic-рекомендацій
- Коректна обробка edge cases: escalation, out of scope, ambiguous-запити
- Дотримання SLA (≤ 10 секунд) та прийнятна вартість запиту
- Реалізована multi-agent crew (≥ 3 агенти) та single-agent baseline на спільному endpoint
- Golden set (15+ задач) прогнаний через LangSmith Experiments на обох архітектурах із side-by-side порівнянням
- Streamlit UI для demo
- Звіт із числовими показниками та обґрунтованим висновком: переваги та недоліки multi-agent архітектури, рекомендації щодо застосування у production

### Стек (рекомендований, не обов'язковий)

- **Backend:** Python 3.11+ · FastAPI · Pydantic
- **Multi-agent framework:** на вибір — LangGraph / CrewAI / AutoGen / OpenAI Agents SDK
- **Single-agent baseline:** plain SDK (Anthropic / OpenAI) + tool_use loop, без фреймворку
- **LLM:** OpenRouter або Anthropic API напряму. Допускається mix моделей (Haiku — для дешевих ролей, Sonnet — для складних)
- **Storage:** SQLite / Postgres із тестовими транзакціями
- **Vector store (long-term memory):** Qdrant / Chroma / FAISS / pgvector — на вибір
- **Observability + Eval:** LangSmith (free cloud tier, native для LangGraph) — обов'язково. Tracing, Datasets та Experiments для порівняння crew vs baseline
- **UI:** Streamlit — обов'язково
